Вступ

Коли проєкт досить великий чи над ним працює не один розробник, а група, виникають питання. Де і як зберігати код? Як отримувати зі сховища актуальну інформацію? Як відшукати стару версію, якщо вона раптом знадобитися. Для розв'язання цих питань і безлічі інших, є системи контролю версій. Вони дозволяють декільком розробникам паралельно працювати над проектом, гарантують, що жоден розробник не видалить код іншого.

Система контролю версій — це програма, що дозволяє розробникам відстежувати і записувати зміни, що вносяться усіма учасниками у файли проекту. Результат може бути об'єднаний в один файл або розбитий на безліч файлів, у будь-якому разі вклад кожного учасника, історія його записів будуть збережені. На сьогодні Git - одна з найпоширеніших систем контролю версій.

Головне завдання курсу - навчання верстанню, git - допоміжний інструмент для автоматизації, зручності здачі домашніх завдань. Ми розглянемо ключові моменти та поняття Git, необхідні для роботи з курсом.

Рекомендуємо ознайомитися з вичерпним підручником від Atlassian у вільний від курсу час. Також у додаткових матеріалах є відеокурс по git російською.

Git

Git (Гіт) — система контролю й управління версіями файлів. Технологія її організації досить специфічна, чимось схожа на зберігання файлів на сервісі Google Drive з тією відмінністю, що усі зміни та версії фіксуються, і для роботи використовується спеціальна програма або термінал.

GitHub

GitHub (Гітхаб) — вебсервіс для розміщення репозиторіїв і спільної розробки. Працює за принципами Git і є хостингом IT-проектів. Надає весь необхідний функціонал для роботи Git. Є подібні сервіси: GitLab і BitBucket.

Репозиторій

Репозиторій (Repository) - каталог файлової системи, в якій містяться файли конфігурації, файли журналів операцій, що виконуються в репозиторії, індекс розташування файлів і сховище, що зберігає найбільш контрольовані версії файлів.

Типи репозиторіїв

  • Локальний репозиторій - розташований на локальному комп'ютері розробника в каталозі. Саме в ньому відбувається розробка та фіксація змін, які надсилаються на віддалений репозиторій.
  • Віддалений репозиторій — міститься на віддаленому сервері. Це загальний репозиторій, в який приходять усі зміни і з якого забираються всі оновлення. Надає можливість спільної роботи над проєктом.

Секції Git-проєкту

  • Робоча директорія (working directory) - є зліпком версії проекту. Файли розпаковуються зі зжатої бази даних у Git-директорії і розміщуються на диску для того, щоб їх можна було змінювати й використовувати.
  • Область підготовлених файлів (staging area) - файл, що міститься у Вашій Git-директорії, у ньому міститься інформація, які зміни попадуть у наступний коміт. Цю область також називаються індексом, проте stage - загальноприйнята назва.
  • Git-директорія (git directory) - місце, у якому Git зберігає метадані і базу об'єктів Вашого проекту. Це найважливіша частина Git, саме вона копіюється при клонуванні репозиторію з іншого комп'ютера.

Стани Git

Стани Git - ключове поняття, яке потрібно засвоїти.. Git має три основних стани, в яких можуть бути файли:

  • Зафіксований (committed) - означає, що файл уже збережений на Вашому локальному репозиторії.
  • Змінений (modified) - файли, що змінилися, але ще не зафіксовані.
  • Підготовлений (staged) - змінені файли, позначенні для приєднання в наступний коміт.

Основи Git

Клонування

Клонування (Clone) — скачування репозиторію з віддаленого сервера на локальний комп'ютер у певний каталог для подальшої роботи з цим каталогом як із репозиторієм.

Під час клонування віддаленого репозиторію на локальному комп'ютері з'явиться папка (її ім'я - назва репозиторію), в якій міститимуться всі файли репозиторію, відбуватиметься зв'язування локальної папки та віддаленого репозиторію. Також у ній з'явиться прихована папка .git, яка є локальним репозиторієм, це службова папка Git.

Завдяки клонуванню розробник може працювати з файлами проєкту: змінювати їх, видаляти, створювати нові. Ці зміни локальні. При завершенні певного етапу роботи розробнику необхідно зробити коміт - зафіксувати виконані зміни.

При клонуванні відбувається копіювання не лише робочої копії. Git отримує копію практично всіх даних, які є на віддаленому сервері. Коли виконується клонування, з сервера забирається кожна версія кожного файлу з історії проєкту.

Коміт

Коміт (Commit) — створення нового зліпка стану файлів проекту, фіксація змін і їх запис у локальний репозиторій. Коміт - мінімальна структурна складова розробки, усі коміти зберігаються в системі контролю версій. Розробник може повернутися до будь-якого коміту.

Щоразу, коли Ви робите коміт ( зберігаєте стан свого проєкту в Git), система запам'ятовує кожен файл і зберігає посилання на цей зліпок. Щоб покращити ефективність, Git не запам'ятовує файли, якщо вони не були змінені, а створює посилання на попередню версію ідентичного файлу, який уже збережений.

Пуш

Пуш (Push) — надсилає всі нові коміти на віддалений репозиторій. Коли Ви хочете додати свої локальні напрацювання на віддалений репозиторій, треба зробити пуш (push). Операція виконується, якщо виконуються дві умови:

  • Якщо ви клонували репозиторій із сервера, на якому Ви маєте право на запис
  • Якщо ніхто з того часу не виконував команду push

Якщо після Вашого клонування, хтось із правом на запис виконав команду push, то Ваш push точно буде відхилений. Щоб команда push спрацювала, необхідно отримати зміни (команда pull), а потім отримані зміни об'єднати з Вашими (merge). Це доцільно лише у командній розробці. Якщо Ви - єдиний учасник процесу, таких ситуацій не буде.

Пул

Пул (Pull) — Отримання останніх змін із віддаленого репозиторію.

Git автоматично дістає і потім зливає дані із віддаленої гілки у Вашу поточну гілку. Усталено операція клонування автоматично налаштовує Вашу локальну гілку master на відстежування віддаленої гілки master на сервері, з якого Ви клонували (мається на увазі, що на віддаленому сервері є гілка master). Виконання pull, як правило, дістає інформацію з сервера, з якого Ви спочатку склонували дані й автоматично намагається злити (merge) їх з кодом, над яким Ви працюєте в даний момент.

Гілки

Гілка (Branch) — паралельна версія репозиторію. Вона міститься в цьому репозиторію, але не впливає на головну версію. Це дозволяє вільно працювати у паралельній гілці. Коли Ви внесли необхідні зміни, Ви можете об'єднати їх із головною версією.

Над проектом зазвичай працюють декілька людей. Розробка полягає в описі певного модуля, виправленні деякої помилки, будь-якій іншій активності, що потребує автономності. Для цього й використовують гілки. Ім'я гілки найчастіше вибирається відповідно до назви модуля чи виправлення.

Процес використання гілки

  • На початку роботи над виправленнями чи новими модулями розробник створює нову гілку з головної.
  • У процесі роботи в разі потреби(при завершенні певної частини роботи) розробник робить коміти в створену гілку.
  • Після завершення роботи з гілкою (повністю розроблений новий модуль чи виправлена помилка), розробник робить злиття створеної гілки з основною.
  • Якщо робота з гілкою у майбутньому не планується, її видаляють. Якщо під час роботи з новою гілкою ніхто з розробників, які працюють над проєктом, не змінював файли, які були змінені в процесі розробки в новій гілці, то злиття відбувається, інакше виникають конфлікти, які необхідно розв'язувати.

Під час розробки є можливість перемикатися між гілками. При цьому в локальній папці, що пов'язана з репозиторієм, показуватиметься та версія файлів, що відповідає поточній гілці.

Майстер

Майстер (Master) — головна чи основна гілка репозиторію. Створюється усталено в момент створення репозиторію.

Мердж

Мердж (Merge) — Злиття змін із якоїсь гілки репозиторію з будь-якою гілкою цього ж репозиторію. Найчастіше це злиття гілки, що містить зміни, з основною.

Конфлікт

У деяких випадках при злитті поточної гілки з головною (master), можуть виникати конфлікти. Причина виникнення завжди одна — певні файли, з якими Ви працювали в створеній гілці, були змінені паралельно в головній також. Унаслідок цього при злитті немає однозначного варіанту зшиття контенту файлів, який не спричинив би втрату даних. У цьому разі розробнику необхідно визначити, які зміни повинні бути залишені.

Розв'язання конфліктів

Розв'язання конфлікту полягає в формуванні кінцевого файлу з двох версій файлів(тієї, яка міститься в головній гілці й тієї, яку Ви хочете туди помістити).

Форк

Форк (Fork) — копія репозиторію. Також можна розглядати як зовнішню гілку для поточного репозиторію. Копія Вашого відкритого репозиторію на GitHub може бути зроблена будь-яким користувачем, згодом він може надіслати зміни у Ваш репозиторій через пулреквест.

Пулреквест

Пулреквест (Pull Request) — запит на злиття форку репозиторію з основним репозиторієм. Пулреквест може бути прийнятий чи відхилений власником репозиторію.

Процес роботи з Git-проектом

  1. Для нового проєкту створюється віддалений репозиторій і клонується на локальну машину. У Вас з'являється точна копія віддаленого репозиторію, яка автоматично з ним зв'язується. Чи створюється локальна папка з проєктом, згодом вона пов'язується з віддаленим репозиторієм.
  2. Робота в головній чи інших гілках.
  3. Ви змінюєте файли у Вашій робочій директорії.
  4. Ви додаєте файли в індекс, у такий спосіб їх зліпки додаються в область підготовлених файлів.
  5. Коли Ви робите коміт, використовуються файли з індексу, даний зліпок зберігається у Вашу Git-директорію.
  6. Якщо Ви працюєте самостійно в гілці, то перед виконанням команди push необхідно злити (merge) гілку в локальний майстер. Якщо розробників декілька, процес роботи з гілками ускладнюється.
  7. Push виконують для оновлення віддаленого репозиторію.

Робота з Git через консоль

З Git можна працювати не тільки за допомогою програм із графічним інтерфейсом, але й з консолі. У веброзробці розуміння Git, вміння працювати з Git через консоль - вимоги.

Таблиця базових команд для роботи з git через консоль (Git є в системі)

Синтаксис Опис
git clone [params] Клонує віддалений репозиторій на комп’ютер, автоматично зв’язує його з локальною версією.
git status Перевіряє наявність змін у стані файлів проекту з моменту останнього коміту. За умови наявності змін виведе в консоль інформацію про файли, в яких є зміни.
git add [file-name чи .] Щоб повідомити Git про необхідність почати відстежувати зміни, внесені у файл, спочатку треба додати їх у staging area, використовуючи git add. При використанні git add . Крапка означає додавання усіх змін.
git reset [filename] Використовується для видалення файлу з-під контролю версій.
git commit -m "коментар" Створює зліпок поточного стану файлів, які під контролем версій. Кожен commit треба описувати коротким, змістовним коментарем про внесені зміни. Кількість комітів необмежена. Отже, якщо нам колись потрібно буде повернутися до минулої версії проекту, ми зможемо "відкотити" його до певного коміту.
git log Виводить журнал змін у репозиторії як список усіх комітів, які були зроблені в хронологічному порядку.
git push Надсилає усі збережені в комітах зміни з локального репозиторію на віддалений. При виконанні git push -u origin master, прапорець -u вказує Git запам'ятати куди ми пушимо (на віддалений репозиторій, origin, у гілку master). Після цього достатньо буде виконувати git push .
git pull Забирає останні зміни з віддаленого репозиторію в локальний, якщо вони зв’язані.
git init Ініціалізує створення локального репозиторію в поточній папці. Створюється прихована папка .git у якій зберігається службова інформація про локальний репозиторій.
git remote add origin [repo-url] Пов’язує віддалений репозиторій з локальним по url. Потрібно, щоб існував віддалений репозиторій. Після виконання команди, можна буде пушити зміни на віддалений репозиторій.

Файл .gitignore

Якщо створити файл .gitignore в своєму репозиторії, Git використовує його, щоб визначити, які файли і каталоги (або цілі шаблони) ігнорувати, перш ніж здійснювати commit. Зазвичай він використовується, щоб уникнути передачі тимчасових файлів з вашого робочого каталогу, які не є корисними для інших співавторів, таких як продукти компіляції, файли налаштувань проекту, створені середовищем розробки і тощо

.gitignore це не розширення файлу, а його ім'я. У провіднику windows створити такий файл не вийде, візьміть .gitignore з вебінару або створіть його при ініціалізації вашого сховища для домашніх робіт.

Правила синтаксису в .gitignore

  • Один рядок - одне правило
  • Порожні рядки ігноруються
  • Коментарі доступні через решітку (#) на початку рядка
  • Символ "/" на початку рядка вказує, що правило застосовується тільки до файлів і папок, які розташовуються в тій же папці, що і сам файл .gitignore
  • Можна використовувати спецсимволи: зірочка (*) замінює будь-яку кількість символів (нуль або більше), питання (?) Замінює від нуля до одного символу. Спецсимволи можна розміщувати в будь-якому місці правила
  • Дві зірочки (**) використовуються для вказівки будь-якої кількості піддиректорій, докладніше дивись нижче в прикладах
  • Знак оклику (!) На початку рядка означає інвертування правила, необхідний для вказівки виключень з правил ігнорування
  • Символ "\" використовується для екранування спецсимволов, наприклад, щоб ігнорувати файл з ім'ям "! Readme! .Txt", потрібно написати таке правило: "\! Readme! .Txt"
  • Для ігнорування всієї директорії, правило повинно закінчуватися на слеш (/), в іншому випадку правило вважається ім'ям файлу

Приклад запису файлу .gitignore

# Ігнор-лист файлів проекту
# Ігнорувати всі файли і директорії, включаючи піддиректорії і файли в них
*

# ---- ФАЙЛИ ----
# Ігнорування по типу файлу, будуть ігноруватися в АБСОЛЮТНО всіх директоріях
# Наприклад /files/data.zip, /server.log, /uploads/users/data/info.xls
*.zip
*.log
*.pdf
